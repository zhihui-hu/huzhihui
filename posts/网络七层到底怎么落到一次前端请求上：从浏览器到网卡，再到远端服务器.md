我以前一直觉得，“前端发一个请求”这句话，挺轻的。

页面里一行：

```ts
const res = await fetch('https://api.example.com/user/profile');
const data = await res.json();
```

看起来就像浏览器把一段 HTTP 发给服务端，然后等一会儿，再把 JSON 拿回来。

真去拆这条链路，味道就完全变了。

这根本不是“浏览器发了个请求”这么简单。它更像是一整个系统临时组了个队：浏览器进程、操作系统网络栈、DNS 解析器、网卡、交换机、路由器、NAT、远端负载均衡、服务端内核，大家配合完这一轮，你才在前端拿到一个 `200 OK`。

所以这篇我不想按教材那种方式，从定义开始背七层。我想反过来，从一次真实的前端请求往下掉，一层一层看。这样网络七层不是抽象图，而是你真能在浏览器、抓包工具、机房设备和内核行为里对上的东西。

## 先把一个误会讲明白：七层模型是讲解视角，不是现实里的七个独立盒子

很多人第一次学网络，会把 OSI 七层学成一张必须死记硬背的表：应用层、表示层、会话层、传输层、网络层、数据链路层、物理层。

这个表当然重要，但它最主要的价值，其实是帮你分清“不同问题分别归谁管”。

现实工程里，互联网更常用的是 TCP/IP 这套分层。很多职责并不会像课本那样切得特别整齐。尤其是表示层、会话层，这两层在真实系统里经常“散”在 TLS、编码、连接复用、认证状态这些机制里面，不会单独站出来说“你好，我就是第五层”。

如果把这次前端请求映射到七层，我更愿意这么看：

| OSI 七层   | 这次请求里真正落到手上的东西                       |
| ---------- | -------------------------------------------------- |
| 应用层     | HTTP、DNS、Cookie、Header、URL、方法和语义         |
| 表示层     | JSON、UTF-8、Brotli/Gzip、TLS 加密前后的数据表示   |
| 会话层     | 连接复用、TLS Session、HTTP Keep-Alive、认证上下文 |
| 传输层     | TCP 或 QUIC，端口、可靠传输、重传、流控、拥塞控制  |
| 网络层     | IP 寻址、路由转发、TTL、分片相关问题               |
| 数据链路层 | 以太网/Wi-Fi、MAC 地址、ARP、交换机转发            |
| 物理层     | 电信号、光信号、无线电波、网线、光模块、射频环境   |

也就是说，七层不是没用。

它很有用。但你别把它理解成“抓一个包时，里面会天然摆着七层小抽屉”。真实系统更像一团叠在一起的实现，七层只是你剖开它时用的刀法。

## 从前端按下回车开始，请求其实先在浏览器里兜了一圈

先拿最常见的场景说。你在地址栏输入一个 URL，或者前端代码里执行一次 `fetch`。

很多人脑子里默认的第一步是“发网络请求”。其实还没那么快。

浏览器通常会先做几件事。

1. 先看这事能不能根本不出网。比如内存缓存、磁盘缓存、`Service Worker`、HTTP 缓存协商、预连接池里有没有现成连接。很多请求压根没碰到网卡，就已经在浏览器内部结束了。
2. 如果必须出网，浏览器会把 URL 解析开。协议是 `https` 还是 `http`，域名是什么，端口是多少，路径和查询参数是什么，这一步决定了后面该走哪套协议栈。
3. 浏览器会问操作系统：这个域名对应哪个 IP？这时候才进入 DNS 解析流程。先查浏览器缓存，再查系统缓存，不行就找本地 DNS 解析器，再由递归解析一路去问根、顶级域、权威 DNS。
4. 拿到目标 IP 之后，还不能立刻发。操作系统会判断这个 IP 是不是和自己在同一网段。如果不是，它真正要找的第一个二层目标，其实是默认网关的 MAC 地址，不是远端服务器的 MAC。
5. 如果网关的 MAC 地址还不知道，就要先发 ARP 请求。这个包很有意思，它在局域网里是广播的，意思大概是：谁是 `192.168.1.1`，把你的 MAC 告诉我。要是走的是 IPv6，这里对应的通常就是邻居发现，也就是 NDP。
6. 到这一步，链路层邻居找到了，传输层才正式开始建连接。`HTTPS` 一般先来 TCP 三次握手，再做 TLS 握手。握手都过了，HTTP 请求头和请求体才有地方放。

这就是为什么我一直觉得，很多前端对“请求发出去了吗”这句话的理解，天然太晚了。

真正的故事在请求发出去之前就已经开始了。

## 真正往下掉的时候，请求会一层一层换壳

我们继续盯着这条 `fetch('https://api.example.com/user/profile')`。

应用层最先看到的是一段 HTTP 语义，比如：

```http
GET /user/profile HTTP/1.1
Host: api.example.com
Accept: application/json
Cookie: sid=abc123
```

如果这是 `HTTPS`，这段内容不会直接裸奔上网。它会先被 TLS 接住。你可以把 TLS 想成一次“先谈规则，再加密运输”的过程：客户端确认支持哪些密码套件，带上 SNI、ALPN 这类握手信息，服务端回证书，客户端验证证书链、域名、有效期，双方协商出会话密钥。

这一步很像 OSI 里的表示层加会话层混在一起了。

因为它既在处理“怎么表示和保护数据”，也在处理“这条安全会话怎么建立、怎么复用、怎么恢复”。

然后才轮到传输层。

如果走的是 HTTP/1.1 或 HTTP/2，大概率底下是 TCP。TCP 不认识“HTTP 请求”这种概念，它只认识字节流。应用层扔给它一串字节，TCP 会把这串字节切成一段一段的 segment，给每段编号，维护发送窗口，等对方 ACK，丢了就重传，快了还要考虑拥塞窗口能不能撑得住。

这块是很多人学网络时第一次真正开始头疼的地方，因为它终于不再是“我发了，你收了”这种线性叙事了。

TCP 处理的是时间、顺序、确认、丢包和拥塞。  
它是在和“不可靠的底层”打交道。

再往下，网络层把 TCP segment 包进 IP packet。这里最关键的是源 IP、目标 IP、TTL、协议号这些字段。路由器不关心你里面是不是 HTTP，也不关心你业务是不是“查用户资料”，它就看目标 IP 该往哪转。

再往下一层，数据链路层把 IP packet 包进 frame。以太网帧头里是源 MAC 和目标 MAC，再带上类型字段、校验信息。注意这里的目标 MAC，通常不是远端服务器的 MAC，而是“下一跳”的 MAC。只要包还没到最终主机，每过一跳，二层头都可能被重写一遍。

最后才到物理层。

这时候它已经不叫“请求”了，它只是比特流，只是电压变化、光脉冲、无线电信号。你在浏览器里看到的 `fetch`，到这里早就没有一点 JavaScript 味儿了。

把这条链压成一行，其实就很清楚：

```txt
HTTP 消息
-> TLS Record
-> TCP Segment
-> IP Packet
-> Ethernet Frame
-> Bits on wire
```

所谓封装，讲白了就是一层套一层壳。

每一层都只对自己那点职责负责。HTTP 不管路由。IP 不管证书。网卡不管你这包是不是登录请求。

系统就是这么被拆开，才撑得住复杂度。

## 交换机、路由器、NAT，中间这些设备干的不是一件事

请求离开你的机器之后，很多人会把中间路径统称成“网络”。这说法不能说错，但太糊。

交换机、路由器、NAT 设备，看的层次不一样，动作也不一样。

交换机主要活在二层。它看的是 MAC 地址，维护的是 MAC 地址表，决定一个 frame 从哪个端口转出去。你在同一局域网里访问旁边一台主机，很多时候主要就是交换机在忙。

路由器活在三层。它看的是 IP 和路由表，判断目标网段该送去哪条链路。你访问公网服务，本质上是把包先交给默认网关，再由一串路由设备一跳一跳往前送。每过一跳，TTL 会减一。减到 0，包就被丢掉，这也是 `traceroute` 能工作的基础之一。

家里那台路由器通常还顺手干了 NAT。你本机可能是 `192.168.0.23` 这种私网地址，公网根本不认。NAT 会把你的源地址和端口改写成路由器自己的公网地址和某个映射端口。等响应回来，再按映射表转回你这台机器。

所以一次看似普通的前端请求，实际上很可能经历的是：

你的浏览器进程把数据交给内核，内核交给网卡，网卡发到交换机，交换机发给路由器，路由器做 NAT，再送到运营商网络，接着一路穿过若干自治系统，最后到达目标服务的边界网关、负载均衡和服务器。

你在控制台里只看到一个 `pending`。

中间其实已经跑过很多设备了。

## 到了服务端，请求会倒着拆壳，然后才轮到业务代码

包到服务端，不是直接进 Nginx，也不是直接进你的 Java、Go、Node 服务。

它要先被网卡收下来。

现代网卡会用 DMA 直接把数据写进内存，内核收到中断或者轮询到数据后，开始沿着协议栈往上拆。先看二层头，再看 IP 头，再把 TCP segment 按序拼回字节流。如果中间有乱序、丢包、重传，也是在内核这一层先处理。

如果是 `HTTPS`，接着要由 TLS 终止点来解密。这个终止点可能在 Nginx，可能在 Envoy，可能在云负载均衡，甚至可能直接在应用进程里。

等到 HTTP 语义终于被还原出来，应用服务器才终于看到：方法是 `GET`，路径是 `/user/profile`，Header 里带了什么，Cookie 或 Token 是什么，请求体又是什么编码。

也就是说，业务代码其实站在很靠上的位置。

你在 Controller 里写的那行 `return userService.profile()`，前面已经有很多层在替你干脏活累活了。顺序重组、重传处理、证书协商、路由转发、邻居发现，这些都不是业务代码的事情，但少任何一层，请求都进不来。

## 前端看到的“慢”，往往不是一个点慢，而是一整条链路在排队

我后来再看浏览器 DevTools 里的 Network Timing，就很难只把它当成一个时间条了。

因为那几个小阶段，其实都对得上底层动作：

- `DNS Lookup` 慢，可能是本地缓存没命中，递归解析慢，或者 DoH/DoT 路径本身有额外开销。
- `Initial connection` 慢，往往是 TCP 握手的 RTT 高，或者 SYN 重传了。
- `SSL` 慢，可能是 TLS 握手、证书链验证、OCSP、密钥协商有额外成本。
- `Request sent` 通常很短，除非上行拥塞或者请求体很大。
- `Waiting (TTFB)` 长，不一定只是服务端业务慢，也可能是前面某个链路阶段卡住，或者服务端在排队。
- `Content Download` 长，可能是响应大，也可能是带宽、拥塞窗口、丢包重传出了问题。

这也是为什么有些问题你明明写的是前端页面，最后却要去查：

- DNS 配置
- 负载均衡健康检查
- 跨地域 RTT
- MTU 和分片
- TCP 重传
- 证书链是否完整

因为“一个请求慢了”这件事，从来都不是纯前端问题。

它是跨层问题。

再多说一句，到了 HTTP/3 这里，故事还会再拐一下。HTTP/3 不再跑在 TCP 上，而是跑在 UDP 之上的 QUIC。这样做的好处是把传输可靠性、拥塞控制和 TLS 1.3 的很多能力揉进了一个用户态协议里，连接建立和多路复用的体验会变。但你会发现，分层思维还是没变。只是“哪一层负责什么”，被重新划了一次。

## 最后收一下：你以为你在学网络七层，其实你在学一次请求是怎么活着到对面的

如果只背定义，网络七层很容易学成一套答题模板。

可一旦你从前端一个真实请求开始往下看，这七层马上就活了。应用层不再只是“HTTP 属于第七层”，而是 URL、Header、Cookie、缓存策略真的在影响请求行为；表示层不再只是一个冷门名词，而是编码、压缩、加密如何改变数据长什么样；会话层也不再悬空，它藏在 TLS Session、连接复用、长连接这些持续状态里；传输层开始变得具体，因为三次握手、重传、滑动窗口、拥塞控制会直接决定你感受到的延迟；网络层开始变得具体，因为 IP 和路由真在决定包怎么走；数据链路层开始变得具体，因为 ARP、MAC、交换机转发决定了它怎么离开你的网卡；物理层最底，也最诚实，网线质量、无线干扰、光模块状态不好，前面那几层都得跟着遭殃。

所以我现在更愿意这样理解“网络层”这三个字。

它不只是七层里的第三层。

很多时候，我们嘴上说“网络层有问题”，实际遇到的是整条网络栈的问题。前端一个请求之所以值得反复拆，不是因为它能帮你背概念，而是因为它把浏览器、操作系统和真实网络世界，硬生生串到了一起。

你看到的是一次 `fetch`。

底下跑的是整个互联网那套老而强的协作机制。
